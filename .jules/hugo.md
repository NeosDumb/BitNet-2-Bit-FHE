## 2024-05-24 - [Scale-Invariant Fused Quantization]
**Learning:** In the `quantize_i2_s` algorithm, the ternary mapping $T(x) \in \{0, 1, 2\}$ traditionally relied on the maximum absolute scale of the tensor $S = \max(|x_i|)$. However, mathematically, for any non-zero element, the condition $x_i \cdot S > 0$ is exactly equivalent to $x_i > 0$ because $S$ is strictly positive. This reveals a critical property: the ternary mapping is structurally scale-invariant. Because of this, the computation of the maximum scale and the mapping process do not need to be sequential. They can be mathematically commuted and fused into a single O(n) pass over the data, completely eliminating the need for intermediate O(n) heap memory allocation (`malloc`/`free`) to store states before packing.
**Action:** Always inspect the mathematical dependencies of a loop. If an operation (like ternary mapping) is invariant to the final result of another operation (like computing the max scale), they should be fused to conserve computational energy (memory bandwidth and allocations).
## 2024-05-25 - [Precision Domain Matching]
**Learning:** In tight numerical loops within `src/ggml-bitnet-mad.cpp` (like `quantize_i2_s`), mixing single-precision float inputs with double-precision arithmetic operations (`fmax`, `fabs`, `1e-6`) creates unnecessary computational overhead. From a physical hardware perspective, promoting single-precision to double-precision inside a tight loop halves SIMD pipeline throughput and introduces conversion delays. Since the input domain is strictly single-precision, there is zero mathematical information gain from using double precision.
**Action:** Always match the precision of mathematical operations to the domain of the input data type. Use native single-precision float routines (e.g., `fmaxf`, `fabsf`) and literals (e.g., `1e-6f`) to avoid implicit conversions, thereby preserving SIMD width and conserving computational energy.
