## 2024-05-24 - [Scale-Invariant Fused Quantization]
**Learning:** In the `quantize_i2_s` algorithm, the ternary mapping $T(x) \in \{0, 1, 2\}$ traditionally relied on the maximum absolute scale of the tensor $S = \max(|x_i|)$. However, mathematically, for any non-zero element, the condition $x_i \cdot S > 0$ is exactly equivalent to $x_i > 0$ because $S$ is strictly positive. This reveals a critical property: the ternary mapping is structurally scale-invariant. Because of this, the computation of the maximum scale and the mapping process do not need to be sequential. They can be mathematically commuted and fused into a single O(n) pass over the data, completely eliminating the need for intermediate O(n) heap memory allocation (`malloc`/`free`) to store states before packing.
**Action:** Always inspect the mathematical dependencies of a loop. If an operation (like ternary mapping) is invariant to the final result of another operation (like computing the max scale), they should be fused to conserve computational energy (memory bandwidth and allocations).
## 2024-05-25 - [Single-Precision Floating-Point Optimization]
**Learning:** Mixing precision types (`float` and `double`) in tight numerical loops is a subtle anti-pattern. Implicit type conversions (upcasting to double-precision) for operations like `fmax` and `fabs` stall SIMD execution pipelines, halving the number of operations per register and artificially restricting computational throughput, despite the input data being inherently single-precision (`const float * src`).
**Action:** Consistently align scalar constants, mathematical functions (`fmaxf`, `fabsf`), and variable definitions with the native precision of the underlying hardware structures (single-precision arrays). Every precision conversion carries an energy and bandwidth cost.
## 2024-05-26 - [Branchless Step Function Quantization]
**Learning:** In tight numerical inner loops, such as ternary quantization mapping `q_val = val > 0 ? 2 : 0` combined with threshold condition `fabs(val) < 1e-6f ? 1`, conditional branches frequently cause pipeline stalls due to CPU branch mispredictions. The mathematical insight is that the ternary output {0, 1, 2} maps perfectly to a combination of arithmetic step functions.
**Action:** Replace sequential `if-else` branching logic with branchless arithmetic mappings: `T(x) = (x > \epsilon) - (x < -\epsilon) + 1`. This reduces execution time and relies on the CPU executing boolean comparisons as numeric values, maintaining vector instruction pipelining and maximizing throughput.
